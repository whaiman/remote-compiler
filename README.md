# RGCC — Remote GCC Compiler System

RGCC is a powerful tool for secure, remote compilation of C/C++ projects. It automatically collects source code and its dependencies (local includes) from a developer machine, securely transfers them to a build server, compiles the code, and returns the finished artifacts back to the client.

## 🌟 Key Features

- **End-to-End Encryption**: AES-256-GCM protects your source code and binaries during transmission.
- **Secure Key Exchange (ECDH + HKDF)**: Dynamic Elliptic-Curve Diffie-Hellman handshake for generating ephemeral session keys without transmitting them over the network.
- **Stateless Session Tickets**: Server memory is protected from DDoS and memory leaks. The server does not maintain session states in its RAM.
- **MITM Protection**: Automatic mixing of a Pre-Shared `auth_token` into the key derivation process to block Man-In-The-Middle attacks.
- **Termux & Mobile Optimized**: Specifically designed to run smooth on Android/Termux and other low-power environments. We deliberately removed heavy dependencies like `FastAPI` and `Pydantic` (which often require Rust compilation during install) to ensure compatibility with virtually any device that can run Python.

> [!NOTE]
> While the server itself is lightweight, compilation speed depends entirely on your project's complexity and the hardware it's running on. Large C++ projects will still take time on mobile CPUs.

## 🛠 1. Server Setup & Execution

Run the following command on the machine acting as your build server (ensure GCC/G++ is installed):

```bash
python -m server
```

**On the first run**, the server will automatically generate a `server_config.yaml` file in the project's root directory containing a secure `auth_token`.

```yaml
# server_config.yaml
server:
  auth_token: <auto-generated-url-safe-base64-token> # <-- Copy this!
  host: 0.0.0.0
  port: 4444

# Cross-compilation & Compiler settings
compilers:
  clang++:
    default_args: ["-fcolor-diagnostics"]
    platforms:
      win64:
        target: "x86_64-w64-mingw32" # Clang --target
        sysroot: "/usr/x86_64-w64-mingw32" # Clang --sysroot
        args: ["-static", "-fuse-ld=lld", "-static-libgcc", "-static-libstdc++"] # Recommended for portable EXEs
  g++:
    platforms:
      win64:
        executable: "x86_64-w64-mingw32-g++" # Use specific cross-compiler binary
...
```

The server dynamically constructs the compilation command based on the client's requested compiler and platform. Using `-static` and `-fuse-ld=lld` is highly recommended for cross-compiling to Windows from Termux/Linux to ensure the output binary is portable and contains all necessary runtime libraries.

## 💻 2. Client Setup & Usage

On the developer machine, a `client_config.yaml` file will be created on the first execution (or you can create it manually).

### ⚠️ IMPORTANT: Configuration Before First Run

For the client to mutually agree on encryption with the server, you must perform two steps in your `client_config.yaml` file:

1. Change `endpoint` to the actual IP address or domain of your build server (keep it as `http://localhost:4444` if testing locally).
2. Insert the `auth_token` that you copied from the `server_config.yaml` file on the server.

Example of a functional `client_config.yaml`:

```yaml
client:
  endpoint: http://192.168.1.50:4444 # Address of the remote build server!
  auth_token: <paste-token-from-the-server-here>
```

### Usage (CLI)

#### 1. Quick Compilation

Compilation is simple—just pass the primary entry point file (`main.cpp` or `main.c`). The CLI will recursively scan for local `#include` paths and package necessary files automatically.

```bash
# Basic compilation
python -m client compile sample/main.cpp

# Specify the output binary name
python -m client compile sample/main.cpp -o my_renderer

# Specify C++ language standard (default is c++17)
python -m client compile sample/main.cpp --std c++20

# Specify target platform (adjusts output extension)
python -m client compile sample/main.cpp --platform win64
```

#### 2. Interactive Mode (`-i`)

If you want to customize settings on the fly without looking up CLI flags, use the interactive mode:

```bash
python -m client compile sample/main.cpp -i
```

The client will prompt you for the compiler, standard, output name, and flags. At the end, it will ask if you want to **save these settings to `build.json`** for future use.

#### 3. Advanced Configuration (`build.json`)

For persistent project settings, you can use a `build.json` file in the project root. It's best created via the `init` command or saved during an interactive session:

```json
{
  "compiler": "clang++",
  "standard": "c++20",
  "entry_point": "sample/main.cpp",
  "output": "main.exe",
  "platform": "win64",
  "flags": ["-Wall", "-O3"],
  "defines": ["DEBUG=1"]
}
```

**Note:** `build.json` only stores your project preferences. Transient data like file lists (`sources`), checksums, and timestamps are automatically calculated during each compilation run to ensure accuracy and security.

#### 4. Other Options

```bash
# Compile only (no linking, similar to -c flag)
python -m client compile sample/main.cpp --compile-only

# Dry run (packages the archive but does not transfer it)
python -m client compile sample/main.cpp --dry-run
```

Once compilation is finished across the network, all resulting artifacts (the compiled binary, GCC logs, and a duration manifest) will be automatically downloaded and extracted into a `compiled/` directory within your current working space.
